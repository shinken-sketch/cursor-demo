# PR 준비 점검

변경된 파일들을 PR 올리기 전 점검해줘.

## 수행 순서

### 1. 테스트 실행

- `npm test`를 실행한다.
- 결과를 요약한다.
  - 통과/실패 여부
  - 실행된 테스트 수
  - 실패한 테스트가 있으면 실패 원인과 관련 파일

### 2. 변경 사항 요약

- 테스트가 **모두 통과한 경우에만** 진행한다.
- `git diff`와 `git status`로 변경된 파일을 확인한다.
- 변경 사항을 **한국어 3줄**로 요약한다.
  - 각 줄은 한 가지 핵심 변경을 담는다.
  - 무엇을, 왜 바꿨는지가 드러나게 쓴다.

### 3. 커밋 메시지 제안

Conventional Commits 형식으로 커밋 메시지를 제안한다.

- **제목 1줄**: `feat:`, `fix:`, `chore:`, `refactor:` 중 적절한 타입으로 시작
- **본문**: 변경 이유와 주요 내용을 2~4문장으로 작성 (한국어)

예시:

```
feat: 이메일 유효성 검사 함수 추가

사용자 입력 이메일 형식을 검증하는 isValidEmail 함수를 추가했다.
기존 validator 모듈과 동일한 패턴으로 구현했으며 단위 테스트를 포함했다.
```

### 4. 리뷰 포인트

마지막에 리뷰어가 **특히 봐야 할 부분**을 짚어준다.

- 로직 변경, 엣지 케이스, API/인터페이스 변경
- 테스트 커버리지가 부족해 보이는 영역
- 회귀 가능성이 있는 부분

## 출력 형식

아래 섹션 순서와 제목을 그대로 사용한다.

```
## 테스트 결과
(요약)

## 변경 사항 요약
1. ...
2. ...
3. ...

## 제안 커밋 메시지
(제목 + 본문)

## 리뷰 포인트
- ...
```

## 주의사항

- 테스트가 실패하면 2~4단계를 건너뛰고, 실패 원인 분석과 수정 방향을 먼저 제시한다.
- 커밋이나 PR을 직접 생성하지 않는다. 점검과 제안만 한다.
- 프로젝트 규칙(`.cursor/rules`)을 따른다.
